# 第0章: 関数型プログラミングの世界へようこそ

> この章は「関数型プログラミングという言葉を初めて聞いた」という人を対象にしています。
> すでに知っている方は第1章から読み始めてください。

---

## 「プログラミングのスタイル」が複数あることを知る

プログラムを書く方法には、いくつかの**スタイル（パラダイム）**があります。

日常の例で考えてみましょう。友達に「コーヒーを淹れてほしい」と頼むとき、2通りの伝え方があります。

**手順を細かく指示する（命令型スタイル）**

> 1. 電気ケトルに水を300ml入れて
> 2. スイッチを入れて90℃になるまで待って
> 3. コーヒーフィルターをセットして
> 4. コーヒー豆を15g量って
> 5. お湯をゆっくり注いで
> 6. 2分待ってカップに移して

**結果だけを伝える（宣言型スタイル）**

> コーヒーを1杯淹れて。

プログラミングでも同じ違いがあります。

- **命令型プログラミング**: 「どのように（How）」処理するかを手順として書く
- **関数型プログラミング**: 「何を（What）」計算するかを式として書く

---

## 身近なアナロジーで理解する

### アナロジー1: 試験の答案 vs 計算の過程

電卓の計算式を考えてください。

```
10 + 20 × 3 = 70
```

この式には「中間の状態」がありません。`20 × 3 = 60` という中間結果は計算されますが、**変数として保存されて後で書き換えられることはない**。これが関数型の考え方です。

一方、命令型では変数を用意して、随時書き換えます。

```
total = 0
total = total + 10   // totalを書き換える
total = total + 60   // また書き換える
```

### アナロジー2: Excelの数式

Excelのセルに `=SUM(A1:A10)` と書くのは宣言型（関数型）のスタイルです。「A1からA10の合計を求めよ」という**意図**を書いており、「どうやって足すか」の手順は書いていません。

Excel VBAでループを書くのが命令型スタイルです。

### アナロジー3: 工場のベルトコンベア

```
原材料 → [洗浄] → [カット] → [加熱] → [包装] → 完成品
```

関数型プログラミングのデータ変換はこのベルトコンベアに似ています。各工程（関数）は独立していて、前の工程の出力が次の工程の入力になります。**工程自体が他の工程の状態を変えることはありません**。

## 関数型プログラミングの3つのコア概念

### 1. 純粋関数（Pure Function）

**同じ入力には、常に同じ出力を返す関数**です。

自動販売機を想像してください。100円コインを入れてボタンAを押せば、必ずジュースAが出てきます。機械の「気分」や「前の人が何を買ったか」は関係ありません。これが純粋関数です。

逆に、同じ操作でも結果が変わる可能性があるのが不純な関数です（在庫切れで別のものが出てくる、など）。

### 2. 不変性（Immutability）

**データを変更するのではなく、変換した新しいデータを作る**考え方です。

出版された本を想像してください。本を「修正」したいとき、既存の本のページを書き換えることはできません。代わりに**改訂版という新しい本**を作ります。元の本はそのまま残ります。

関数型プログラミングでは、データも同じように扱います。

### 3. 関数合成（Function Composition）

**小さな関数を組み合わせて大きな処理を作る**手法です。

Lego（レゴ）を思い浮かべてください。シンプルなブロック（関数）を組み合わせることで、複雑な構造を作れます。各ブロック（関数）は独立していて、別の組み合わせに使い回せます。

## なぜ関数型プログラミングを学ぶのか？

### 理由1: バグが減る

不変データと純粋関数を使うと、「どこかで変数が書き換えられていた」というバグが起きにくくなります。関数が外部の状態に依存しないため、**関数の動作を局所的に理解できます**。

### 理由2: テストが楽になる

純粋関数は「入力と出力だけ」を気にすればよいため、テストが簡単です。データベースのモックを用意したり、グローバル変数を初期化したりする必要がありません。

```rust
// テストしやすい純粋関数の例
fn double(x: i32) -> i32 { x * 2 }

// テストは入力と出力を確認するだけ
assert_eq!(double(3), 6);
assert_eq!(double(-1), -2);
```

### 理由3: 並列処理と相性が良い

複数のスレッドで同じデータを同時に読んでも、誰も書き換えないなら競合は起きません。不変データは**スレッドセーフ**です。

### 理由4: コードが「読める」

```rust
// 命令型: 「何をしているか」を読み解く必要がある
let mut result = Vec::new();
for item in &orders {
    if item.price > 1000 {
        result.push(item.name.clone());
    }
}

// 関数型: 「意図」が直接読める
let result: Vec<_> = orders.iter()
    .filter(|item| item.price > 1000)
    .map(|item| item.name.clone())
    .collect();
```

後者は「1000円超の注文の名前を集める」という意図が一目でわかります。

## Rust と関数型プログラミングの関係

Rust は**純粋な関数型言語ではありません**。しかし、関数型プログラミングのエッセンスを積極的に取り入れています。

| 関数型の概念 | Rust での対応 |
|------------|-------------|
| 不変性がデフォルト | `let` はデフォルトで不変。`let mut` が必要なのは例外 |
| 純粋関数 | 所有権システムが副作用の管理を助ける |
| 関数を値として扱う | クロージャ、`Fn`/`FnMut`/`FnOnce` トレイト |
| データ変換 | `Iterator` の `map`/`filter`/`fold` |
| 代数的データ型 | `enum`（`Option<T>`, `Result<T, E>` など） |
| パターンマッチング | `match` 式 |

Rust でコードを書くとき、関数型スタイルを意識すると**安全で読みやすいコード**が自然と書けるようになります。

## よくある疑問

**Q. 関数型プログラミングは Haskell のような特殊な言語でしかできないの？**

A. いいえ。Rust・Python・JavaScript・Scala・Kotlin など、多くのモダンな言語が関数型の機能を持っています。この資料では Rust を使いますが、学んだ考え方は他の言語でも活きます。

**Q. 命令型プログラミングはダメなの？**

A. そんなことはありません。どのスタイルにも適した場面があります。実際の開発では命令型と関数型を状況に応じて使い分けます。ただし、関数型のツールを知っておくと**選択肢が広がります**。

**Q. 難しそう。Monad とか Functor とか聞いたことがある...**

A. これらは発展的な概念（第8章以降）で、知らなくてもほとんどのことはできます。第1〜4章の基礎をしっかり身に付ければ、日常の Rust コーディングで十分な恩恵が得られます。
